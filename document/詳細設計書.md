## 1. テーブル定義

> 顧客（customers）・商品（products）テーブルは `基本設計書.md` を参照。

### orders（受注ヘッダ）

> 1件の受注全体の情報を管理するテーブル。
> `total_amount`（合計金額）は明細保存後にサーバーサイドで計算・更新する。

| カラム名 | 型 | 制約 | 備考 |
|---------|---|------|-----|
| id | BIGINT | PK, AUTO INCREMENT | 自動採番 |
| customer_id | BIGINT | NOT NULL, FK → customers.id | 受注した顧客 |
| order_date | DATE | NOT NULL | 受注日 |
| status | VARCHAR(20) | NOT NULL | PENDING / PROCESSING / SHIPPED / COMPLETED |
| total_amount | INTEGER | NOT NULL, DEFAULT 0 | 合計金額（明細保存後に更新） |
| created_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | 登録日時（自動設定） |

### order_details（受注明細）

> 1件の受注に含まれる商品ごとの明細情報を管理するテーブル。
> `unit_price` は登録時点の商品単価をコピーして保存する（価格の履歴保持）。

| カラム名 | 型 | 制約 | 備考 |
|---------|---|------|-----|
| id | BIGINT | PK, AUTO INCREMENT | 自動採番 |
| order_id | BIGINT | NOT NULL, FK → orders.id | どの受注の明細か |
| product_id | BIGINT | NOT NULL, FK → products.id | どの商品か |
| quantity | INTEGER | NOT NULL, CHECK >= 1 | 数量（1以上） |
| unit_price | INTEGER | NOT NULL | 受注時点の単価（商品マスタからコピー）|
| subtotal | INTEGER | NOT NULL | unit_price × quantity（小計）|

### users（ユーザー）

> ログイン認証に使用するユーザー情報を管理するテーブル。
> `user` は PostgreSQL の予約語のため、テーブル名は `users` とする。

| カラム名 | 型 | 制約 | 備考 |
|---------|---|------|-----|
| id | BIGINT | PK, AUTO INCREMENT | 自動採番 |
| username | VARCHAR(50) | NOT NULL, UNIQUE | ログイン用ユーザー名 |
| password | VARCHAR(255) | NOT NULL | BCrypt ハッシュ化済みパスワード |

## 3. クラス構成

> 顧客マスタのクラス構成は `基本設計書.md` を参照（商品マスタも同様の構成）。

### 受注管理

```
controller/
  OrderController.java     … 受注の登録・一覧・詳細のルーティング

service/
  OrderService.java        … 受注登録トランザクション・検索ロジック

repository/
  OrderRepository.java     … orders テーブルの操作（ステータス・日付範囲検索クエリ含む）
  OrderDetailRepository.java  … order_details テーブルの操作

entity/
  Order.java               … orders テーブルに対応するエンティティ
  OrderDetail.java         … order_details テーブルに対応するエンティティ

form/
  OrderForm.java           … 受注登録フォームの入力値（明細リスト含む）
  OrderDetailForm.java     … 受注明細1行分の入力値
```

### 認証・アカウント管理

```
controller/
  AuthController.java              … /register の GET/POST のみ担当（/login は Spring Security が処理）

service/
  CustomUserDetailsService.java    … UserDetailsService の実装（Spring Security から呼ばれる）

repository/
  UserRepository.java              … users テーブルの操作

entity/
  User.java                        … users テーブルに対応するエンティティ

form/
  RegisterForm.java                … アカウント登録フォームの入力値

config/
  SecurityConfig.java              … Spring Security の認証・認可設定（★提供済み・変更不要）
  WebConfig.java                   … サイドバーのアクティブ表示設定（★提供済み・変更不要）
```

## 4. バリデーション仕様

> 顧客・商品のバリデーション仕様は `基本設計書.md` を参照。

### 受注ヘッダ（OrderForm）

| フィールド | バリデーション |
|----------|------------|
| customer_id | 必須（`@NotNull`） |
| order_date | 必須（`@NotNull`） |

> `status` と `total_amount` はサーバーサイドで自動設定するため、フォームには含まない。

### 受注明細（OrderDetailForm）

| フィールド | バリデーション |
|----------|------------|
| product_id | 必須（`@NotNull`） |
| quantity | 必須（`@NotNull`）、1以上（`@Min(1)`） |

> `unit_price` と `subtotal` はサーバーサイドで計算するため、フォームには含まない。

### アカウント登録（RegisterForm）

| フィールド | バリデーション |
|----------|------------|
| username | 必須（`@NotBlank`） |
| password | 必須（`@NotBlank`） |

> ユーザー名の重複チェックはバリデーションアノテーションではなく、Controller または Service で `userRepository.existsByUsername()` を使って行う。

## 5. エラー処理方針

> 基本方針は `基本設計書.md` を参照。以下は受注系・認証系の補足。

| 状況 | 対応 |
|-----|-----|
| 存在しない受注IDへのアクセス | `ResourceNotFoundException` をスロー → `/error` へ遷移 |
| 受注登録時に存在しない顧客IDを指定 | `ResourceNotFoundException` をスロー → ロールバック後に `/error` へ遷移 |
| 受注登録時に存在しない商品IDを指定 | `ResourceNotFoundException` をスロー → ロールバック後に `/error` へ遷移 |
| ユーザー名が重複するアカウント登録 | 同一画面に戻り、エラーメッセージを表示する（例: 「このユーザー名はすでに使用されています」） |

## 6. シーケンス図

### 6.1 受注登録（POST /orders）

```
ブラウザ
  │  POST /orders（OrderForm）
  ▼
OrderController.create(form, bindingResult)
  │  ① バリデーションチェック
  │    エラーあり → order/form.html を返す（エラーメッセージ表示）
  │    エラーなし → OrderService.create(form) を呼ぶ
  ▼
OrderService.create(form)  ← @Transactional（ここから全処理が1トランザクション）
  │
  │  ② customerRepository.findById(customerId)
  │    → 見つからない場合: ResourceNotFoundException をスロー → ロールバック
  │
  │  ③ orders テーブルに INSERT（total_amount = 0 で保存）
  │    → orderRepository.save(order)
  │    → order.id が確定する（明細の外部キーに使用）
  │
  │  ④ 明細ごとにループ処理:
  │    productRepository.findById(productId)
  │    → 見つからない場合: ResourceNotFoundException をスロー → ロールバック
  │    subtotal = product.unit_price × detail.quantity
  │    orderDetailRepository.save(detail)
  │    totalAmount += subtotal
  │
  │  ⑤ orders テーブルの total_amount を UPDATE
  │    order.setTotalAmount(totalAmount)
  │    orderRepository.save(order)
  │
  │  ⑥ 全処理正常終了 → コミット
  ▼
OrderController
  │  redirect:/orders
  ▼
ブラウザ（受注一覧画面）
```

> **なぜ orders を2回 save するのか？**
> 明細（order_details）には `order_id`（外部キー）が必要です。
> 最初の `save` で `order.order_id` が確定するため、ヘッダを先に保存してから明細を保存する必要があります。
> 明細保存完了後に `total_amount` を確定値で更新するため、ヘッダは合計2回保存されます。

### 6.2 ログイン認証（POST /login）

```
ブラウザ
  │  POST /login（username, password）
  ▼
Spring Security（UsernamePasswordAuthenticationFilter）
  │  ① username・password を受け取る
  ▼
CustomUserDetailsService.loadUserByUsername(username)
  │  ② userRepository.findByUsername(username)
  │    → 見つからない場合: UsernameNotFoundException をスロー → ログイン失敗
  │    → 見つかった場合: UserDetails を組み立てて返す
  ▼
Spring Security
  │  ③ 入力パスワード（平文）と DB のハッシュを BCrypt で照合
  │    → 不一致: ログイン失敗 → /login?error にリダイレクト
  │    → 一致: ログイン成功 → セッション生成
  ▼
ブラウザ（/customers にリダイレクト）
```

## 7. セキュリティ設計

### 7.1 認証・認可の設定（SecurityConfig.java）

以下の設定がスキャフォールドとして提供されている（変更不要）。

| 設定項目 | 内容 |
|---------|-----|
| 未認証でアクセス可能なURL | `/login`, `/register` |
| それ以外のURL | 認証必須（未認証の場合は `/login` にリダイレクト） |
| ログイン成功後のリダイレクト先 | `/customers` |
| ログアウト成功後のリダイレクト先 | `/login?logout` |
| パスワードエンコーダー | `BCryptPasswordEncoder`（Bean として登録済み） |

### 7.2 パスワードの取り扱い

- **登録時**: `passwordEncoder.encode(rawPassword)` でハッシュ化してから DB に保存する
- **認証時**: Spring Security が自動的に BCrypt 照合を行う（開発者が直接比較する必要はない）
- **禁止事項**: パスワードを平文のまま DB に保存してはいけない

### 7.3 CustomUserDetailsService の役割

Spring Security のログイン処理から呼び出され、ユーザー名をキーに DB からユーザー情報を取得して返すクラス。
`UserDetailsService` インターフェースを実装し、`loadUserByUsername` メソッドをオーバーライドする。

```java
// 実装のポイント（穴埋めの参考）
@Override
public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
    // 1. userRepository.findByUsername(username) でユーザーを検索する
    // 2. 見つからない場合 → UsernameNotFoundException をスロー
    // 3. 見つかった場合 → User.withUsername(...).password(...).roles("USER").build() で返す
    //    ※ password() には DB に保存済みのハッシュをそのまま渡す
}
```

## 8. フェーズB〜Eのログインについて

`SecurityConfig.java` はフェーズB の開始時点からスキャフォールドに含まれているため、
`CustomUserDetailsService`（フェーズE で実装）を作成するまでの間も認証が有効になっている。

フェーズB〜E の期間は、Spring Boot が起動時に自動生成する一時パスワードを使用してログインする。

```
# アプリ起動時にコンソールに表示されるメッセージ（例）
Using generated security password: a1b2c3d4-e5f6-7890-abcd-ef1234567890
```

| 項目 | 値 |
|-----|---|
| ユーザー名 | `user` |
| パスワード | 起動時にコンソールに表示された値（アプリ再起動ごとに変わる） |

フェーズE で `CustomUserDetailsService` を実装すると、DB に登録したアカウントでログインできるようになる。
