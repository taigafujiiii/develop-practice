# 要件定義書

**システム名**: 受発注・顧客管理システム（training-order-system）
**作成日**: 2026-04-01
**バージョン**: 1.0

---

## 1. システム概要

### 1.1 目的

本システムは、中小規模の BtoB 企業における「顧客管理・商品管理・受注業務」をシステム化し、
手作業による管理業務の非効率・ミスを解消することを目的とする。

### 1.2 背景・課題

現状、顧客情報・受注情報は Excel ファイルで管理されているが、以下の課題が発生している。

| # | 課題 | 具体的な問題 |
|---|-----|-----------|
| 1 | バージョン管理の困難さ | 複数の担当者が同一ファイルを更新するため、最新版の管理が煩雑 |
| 2 | 入力ミスの発生 | 単価・数量の手計算によるミス、記入漏れが散見される |
| 3 | 検索性の低さ | 顧客名・受注日での絞り込みに時間がかかる |
| 4 | 受注時の単価管理 | 商品単価が変わった場合、過去の受注金額が変動してしまう |

本システムの導入により、これらの課題をすべて解消する。

### 1.3 対象ユーザー

| ユーザー | 主な利用シーン |
|---------|-------------|
| 営業担当者 | 顧客登録・受注登録・受注状況の確認 |
| 管理者 | 商品マスタ管理・全データの参照 |

## 2. 機能要件

### 2.1 顧客管理機能

| 機能ID | 機能名 | 概要 |
|-------|-------|-----|
| F-01 | 顧客登録 | 顧客の基本情報（名称・電話番号・住所・メールアドレス等）を登録する |
| F-02 | 顧客一覧・検索 | 登録済み顧客を一覧表示する。顧客名での部分一致検索が可能 |
| F-03 | 顧客編集 | 登録済み顧客情報を更新する |
| F-04 | 顧客削除 | 登録済み顧客を削除する |

### 2.2 商品管理機能

| 機能ID | 機能名 | 概要 |
|-------|-------|-----|
| F-05 | 商品登録 | 商品の基本情報（名称・分類・単価・在庫数）を登録する |
| F-06 | 商品一覧・フィルター | 登録済み商品を一覧表示する。カテゴリによる絞り込みが可能 |
| F-07 | 商品編集 | 登録済み商品情報を更新する |
| F-08 | 商品削除 | 登録済み商品を削除する |

### 2.3 受注管理機能

| 機能ID | 機能名 | 概要 |
|-------|-------|-----|
| F-09 | 受注登録 | 顧客・受注日・受注明細（商品・数量）を1操作で一括登録する |
| F-10 | 受注一覧・フィルター | 登録済み受注を一覧表示する。ステータス・受注日の範囲指定での絞り込みが可能 |
| F-11 | 受注詳細 | 受注ヘッダ情報と全明細行を1画面で表示する |

### 2.4 認証・アカウント管理機能

| 機能ID | 機能名 | 概要 |
|-------|-------|-----|
| F-12 | ログイン | ユーザー名・パスワードによる認証を行う |
| F-13 | ログアウト | セッションを破棄してログアウトする |

## 4. 業務フロー

### 4.1 受注登録の流れ

```
① 担当者がシステムにログインする
        ↓
② 受注一覧画面から「新規登録」を選択する
        ↓
③ 顧客をプルダウンで選択し、受注日を入力する
        ↓
④ 受注明細（商品・数量）を1件以上追加する
        ↓
⑤ 「登録」ボタンを押す
        ↓
⑥ システムが受注ヘッダと明細を同時に保存する
   ※ 合計金額はシステムが自動計算する（単価 × 数量 の合計）
   ※ 途中でエラーが発生した場合は全件ロールバックされる
        ↓
⑦ 受注一覧画面に遷移し、登録した受注が表示される
```

### 4.2 受注ステータスの遷移

受注の処理状況は以下の4段階で管理する。

```
PENDING（受付済）→ PROCESSING（処理中）→ SHIPPED（出荷済）→ COMPLETED（完了）
```

| ステータス | 意味 |
|----------|-----|
| PENDING | 受注を受け付けたが、まだ処理が始まっていない状態 |
| PROCESSING | 受注の処理（在庫確認・ピッキング等）が進行中の状態 |
| SHIPPED | 商品が出荷された状態 |
| COMPLETED | 受注が完了した状態 |

## 5. 用語定義

| 用語 | 意味 |
|-----|-----|
| 受注ヘッダ | 1件の受注全体の情報（顧客・受注日・ステータス・合計金額）。orders テーブルに格納 |
| 受注明細 | 受注に含まれる商品ごとの明細情報（商品・数量・単価・小計）。order_details テーブルに格納 |
| マスタ | 顧客・商品など、業務の基本情報を管理するテーブル（頻繁に変わらないデータ） |
| 物理削除 | DB からレコードを完全に削除する方式（本システムはすべて物理削除） |
| スキャフォールド | 受講生が実装するための骨格コード。`work/training-order-system/` に HTML テンプレート・SecurityConfig.java・application.properties が提供済み |

## 6. 制約・前提条件

1. **受注の変更・削除禁止**
   受注登録後は、ヘッダ・明細ともに編集・削除を行わない。
   ステータスの変更は将来的な拡張として想定するが、本システムの実装対象外とする。
   （業務上、受注記録は証跡として保持するため物理削除も行わない）

2. **受注時の単価保存**
   受注登録時の単価は、商品マスタの `unit_price` をコピーして `order_details.unit_price` に保存する。
   これにより、後から商品マスタの単価が変更されても、受注時の金額が保持される。

3. **合計金額の計算**
   `orders.total_amount` はサーバーサイドで計算する（フォームから受け取らない）。
   計算式: `Σ（unit_price × quantity）`

4. **認証の適用範囲**
   `/login` と `/register` は未認証ユーザーもアクセス可能。
   それ以外の全URLはログイン必須とする。

5. **商品マスタの登録日時**
   商品（products）テーブルは登録日時（`created_at`）を管理しない。
   顧客情報と異なり、商品マスタは登録日よりも現在の状態（名称・単価・在庫数）の管理が主目的のため。

6. **アカウント管理**
   本システムにはユーザー権限の区分（管理者・一般）は設けない。
   全ユーザーが同等の操作権限を持つ（`ROLE_USER` 固定）。
